# 차지백(Chargeback) 분쟁처리 어시스턴트 — 업무 규칙

> 이 파일은 **현업 담당자가 읽고 직접 고치는** 업무 규칙서입니다.
> 코드가 아니라 "이 도구가 어떤 업무를, 어떤 규칙으로 도와주는가"를 한국어로 적습니다.
> Claude Code는 이 파일을 항상 먼저 읽고, 여기 적힌 규칙대로 일합니다.

---

## 1. 이 도구가 돕는 업무 (발급사 입장의 1CB 이의신청)

이 도구는 **발급사(Issuer) 입장의 차지백(OB, Outbound)** 을 돕습니다.
즉, **하나카드가 자사 카드홀더를 대리**해, 카드홀더가 받은 분쟁(환불 미이행·청구금액 상이·이중청구 등)을
**처음 청구(1CB, First Chargeback)를 개시** 하는 업무입니다.
핵심 산출물은 **1CB 이의신청 서류(Cardholder Letter + 증빙 묶음)** 입니다.

전체 차지백 프로세스 중 **자료 분석**과 **사유코드·서류 작성** 단계만 돕습니다.

| 단계 | 업무 | 이 도구의 역할 |
|------|------|----------------|
| (1) 접수 | 카드홀더가 raw 증빙 제출 | ❌ 범위 밖 (기존 채널) |
| **(2) 자료 분석** | 비정형 고객 제출 자료 분석·정리 | ✅ 핵심사실·타임라인·쟁점 추출 |
| **(3) 사유코드 + 1CB 레터** | VISA 규정 기반 사유코드 확인, **발급사 제출요건** 충족 체크, **1CB 영문 레터** 초안 작성 | ✅ 후보 제시·초안 작성 |
| (4) 제출 이후 | Pre-Arbitration·중재 | ❌ 범위 밖 |

### 사유코드와 근거 — 발급사 제출요건(Dispute Processing Requirements)

발급사가 1CB를 개시할 때는 사유코드별 **발급사 제출 필수서류** 요건을 충족해야 합니다.
이 도구가 주로 다루는 사유코드와 근거 섹션은 다음과 같습니다.

| 사유코드 | 의미 | 발급사 제출요건 근거 |
|----------|------|----------------------|
| **13.6 Credit Not Processed** (취소/환불 미이행) ⭐ 핵심 | 환불·취소(void/cancelled) 영수증을 받았으나 credit이 처리되지 않음. 영수증 일자 기준 **15일 대기 후 개시·120일 이내** 처리. | **§11.10.7.5** (정의 §11.10.7.1) |
| 12.5 Incorrect Amount (청구금액 상이) | 거래금액 오류·자릿수 전치 등. Dispute 금액은 차액으로 한정. | §11.9.4.5 (정의 §11.9.4.1) |
| 12.6 Duplicate Processing / Paid by Other Means (이중청구/타수단 결제) | 동일 거래 중복 처리 또는 타수단 기결제. | §11.9.5.5 (정의 §11.9.5.1) |

- **13.6에 집중**합니다(승소 사례·자료가 풍부). 12.5/12.6은 보조적으로 다룹니다.
- 13.6 근거를 인용할 때는 반드시 **§11.10.7.x 계열**을 씁니다(§11.10.2.x는 13.6이 아니므로 사용하지 않음).
- 13.6 제출요건의 핵심: ①Credit Transaction Receipt 사본, ②void 표기 거래영수증 사본, ③credit이 발생해야 함을 입증하는 기타 기록 중 **하나**.
  즉 "환불·취소가 확정됐다는 증거 + 그럼에도 입금이 안 됐다는 정황"이 필요합니다.

## 2. 핵심 원칙 (반드시 지킴)

1. **사람이 최종 결정합니다 (Human-in-the-loop 필수).**
   - AI는 **초안·후보·제안만** 만듭니다.
   - **사유코드 최종 선택**과 **첨부서류 최종 승인**은 반드시 현업 담당자가 합니다.
   - AI 출력에는 항상 "확정 전 담당자 검토 필요"가 표시되어야 합니다.

2. **VISA 규정이 최우선 근거입니다.**
   - 사유코드, 증빙 가능/불가, 발급사 제출 필수서류는 모두 `rules/` 폴더의 VISA 분쟁 규정을 근거로 합니다.
   - 규정에 없는 내용을 지어내지 않습니다. 근거가 약하면 "규정 확인 필요"라고 표시합니다.
   - 규정과 사례가 충돌하면 **규정이 우선**입니다. 승소 사례는 "어떻게 썼나(문구·구성)"의 참고일 뿐, "무엇이 필수냐(요건)"는 규정이 정합니다.

3. **고객 서류는 매우 비정형이고, 이미지 입력이 많습니다.**
   - 메신저/채팅 스크린샷, 가맹점 화면 캡처(예: VOID 표기), 현지 매장 취소 영수증 사진, 호텔 청구서 PDF,
     경찰 신고서(외국어 다페이지 PDF), 자필/타이핑 진술서 등 형식이 제각각입니다.
   - **이미지·스크린샷도 직접 읽어(비전 처리)** 사실을 추출합니다. 형식에 의존하지 말고
     "무슨 일이 언제 일어났나(사실·타임라인)"를 뽑아냅니다.
   - 외국어·수기·저품질 스캔이 흔하므로, 추출이 안 되면 날조하지 말고 부분결과 + 수동 보완 경로를 표시합니다.

4. **개인정보(PII) 취급 — POC 전제.**
   - 이 골격은 검증용(POC)이라 **익명 더미 데이터만** 사용합니다.
   - 가짜 이름, 카드번호 `1111-1111-1111-1111` 등 마스킹 데이터만 넣습니다.
   - 입력 증빙에 실명·이메일·전화·카드번호·여권/신분번호가 노출될 수 있으므로, 산출물 작성 전 **마스킹 필수**.
     미상 식별값은 날조 없이 `[CONFIRM]`/`[MASKED]` 로 표기합니다.
   - **실제 고객 PII는 절대 넣지 않습니다.** (실 운영 전 별도 보안 검토 필요)

5. **언어 규칙.**
   - 업무 규칙·설명·분석 결과·체크리스트 → **한국어**
   - **1CB 카드홀더레터(Cardholder Letter) 본문만 영문**으로 작성합니다 (VISA 제출용).

## 3. 업무 흐름 (자료 분석 → 사유코드 + 발급사 제출요건 → 1CB 영문 레터)

```
[카드홀더 제출 raw 증빙 (스크린샷·사진·PDF·진술서 등 비정형·이미지)]
        │
        ▼
 ① 자료 분석   ── 비정형·이미지 증빙 읽기 → 요약·핵심사실·타임라인·쟁점
        │
        ▼
 ② 사유코드 + 발급사 제출요건 체크
        │         ── 사실 + VISA 규정(rules/) → 사유코드 확인 + 근거(13.6 = §11.10.7.x)
        │            + 발급사 제출 필수서류 충족 여부 체크리스트
   [현업 담당자: 사유코드 최종 확인]   ← 사람 결정 지점 (HITL)
        │
        ▼
 ③ 1CB 영문 레터 초안
                  ── 확정 사유코드 + 사실 → Cardholder Letter(영문) 초안 + 증빙 목록
   [현업 담당자: 서류 최종 승인]        ← 사람 결정 지점 (HITL)
```

> **이 흐름을 수행하는 스킬은 미리 만들어져 있지 않습니다.**
> Lab 3에서 참가자가 **skill-creator로 OB 차지백 스킬을 직접 만듭니다.**
> 스킬에는 자료 분석 방법, 사유코드별 발급사 제출요건, 1CB 영문 레터의 구조·톤이 담깁니다.
> (스킬과 그 외 도구는 아래 `.claude/` 폴더에 들어가며, 워크숍 시작 시점에는 비어 있습니다.)

### 1CB Cardholder Letter — 구조·톤 (참고)

- 영문 발급사 표준 폼: 제목("CARDHOLDER LETTER") + 마스킹된 계좌번호 + 거래 표(ARN·거래일·금액·가맹점명) +
  사유 체크리스트(해당 사유 1개 체크, 13.6이면 "Credit not processed") + Comment 자유서술(5–6줄) + 서명/ID 라인.
- Comment 톤: **1인칭·사실 중심·감정 배제.** 시간순으로 짧은 문장. 13.6 전형 구조 =
  ①환불/취소 대상 거래 명시 → ②가맹점이 환불/취소를 확인한 사실 → ③그럼에도 입금 안 됨 → ④연락기록 보유 진술 → ⑤조속한 환불 요청.
- 추측·비난·과장 없이 "환불/정정 요청"으로 단정하게 마무리.

## 4. 이렇게 사용하세요 (실행 예시)

> **코딩은 필요 없습니다.** 터미널에서 이 폴더(`workshop/mvp/`)에서 `claude` 를 실행한 뒤,
> 평소 쓰는 한국어 문장으로 한 단계씩 시키면 됩니다.
> 각 단계 결과는 **사람이 직접 읽고 고친 다음** 다음 단계로 넘어갑니다 (사람이 최종 결정 — HITL).

**시작:** 터미널에서 아래를 입력합니다.

```
cd workshop/mvp
claude
```

> ⚠️ 아래 단계는 **Lab 3에서 직접 만든 OB 차지백 스킬이 있을 때** 가장 잘 동작합니다.
> 스킬을 만들기 전에는 Claude가 이 CLAUDE.md와 `rules/` 만 보고 일반적으로 처리합니다.

**1단계 — 자료 분석**

```
sample-data/case-anon-ob-01 폴더의 증빙(스크린샷·사진·PDF 포함)을 읽어서
핵심 사실·타임라인·쟁점을 한국어로 정리해줘.
```

→ 결과로 "무슨 일이 언제 일어났는지"가 정리됩니다.
→ **사람이 검토:** 사실관계·날짜·금액이 맞는지 확인하고 틀린 부분은 고쳐서 알려줍니다.

**2단계 — 사유코드 확인 + 발급사 제출요건 체크**

```
방금 정리한 사실로 VISA 사유코드를 확인하고(특히 13.6 여부),
발급사 제출 필수서류 충족 여부를 근거(§11.10.7.x 등)와 함께 체크리스트로 제시해줘.
```

→ 결과로 사유코드(예: 13.6)와 근거, 발급사 제출요건 충족/부족 항목이 나옵니다.
→ **사람이 최종 결정:** 사유코드를 **확정**합니다. (← 사람 결정 지점)

**3단계 — 1CB 영문 카드홀더레터 초안**

```
확정한 사유코드 13.6으로 1CB 영문 Cardholder Letter 초안을 작성하고,
첨부할 증빙 목록도 함께 정리해줘.
```

→ 결과로 영문 Cardholder Letter 초안과 증빙 목록이 `sample-data/case-anon-ob-01/output/1cb-draft.md` 에 저장됩니다.
→ **사람이 최종 승인:** 모르는 값은 `[CONFIRM]`/`[MASKED]` 로 표시되니, 담당자가 채우고 승인한 뒤 제출합니다. (← 사람 결정 지점)

> 💡 한 문장으로 이어서 시켜도 되지만, **워크숍에서는 단계마다 멈춰 사람이 확인**하는 방식을 권장합니다.
> 사유코드 선택이 가장 중요한 판단이기 때문입니다.
