# 차지백(Chargeback) 분쟁처리 어시스턴트 — 업무 규칙

> 이 파일은 **현업 담당자가 읽고 직접 고치는** 업무 규칙서입니다.
> 코드가 아니라 "이 도구가 어떤 업무를, 어떤 규칙으로 도와주는가"를 한국어로 적습니다.
> Claude Code는 이 파일을 항상 먼저 읽고, 여기 적힌 규칙대로 일합니다.

---

## 1. 이 도구가 돕는 업무 (인바운드 차지백 — 매입사 방어 우선)

이 도구는 **인바운드(Inbound) 차지백**을 우선 돕습니다.
인바운드란 **하나카드가 매입사(Acquirer)** 인 건, 즉 우리 가맹점이 카드홀더(상대 발급사)로부터 분쟁을 당한 건입니다.
이때 우리 업무는 **가맹점을 방어**하는 것이고, 핵심 산출물은 **재반박(Acquirer Dispute Response / Representment)** 입니다.

전체 차지백 프로세스 중 **2단계와 3단계**만 돕습니다.

| 단계 | 업무 | 이 도구의 역할 |
|------|------|----------------|
| (1) 접수 | VISA 접수내역(분쟁) 수신 | ❌ 범위 밖 (기존 시스템) |
| **(2) 서류 분석** | 비정형 가맹점 자료·접수내역 분석·정리 | ✅ 핵심사실·타임라인·쟁점 추출 |
| **(3) 사유코드+재반박** | VISA 규정 기반 사유코드 확인, (방향별) 필수증빙 안내, **재반박서(영문) 초안 작성** | ✅ 후보 제시·초안 작성 |
| (4) 제출 이후 | Pre-Arbitration·중재 | ❌ 범위 밖 |

### 산출물 두 가지 — 재반박(우선) vs 1CB(보조)

| 산출물 | 우리 입장 | 화자 | 무엇을 하나 | 증빙 출처 |
|--------|-----------|------|-------------|-----------|
| **재반박 (Acquirer Dispute Response)** ⭐기본 | **매입사** | 매입사가 가맹점 대리 | 카드홀더의 차지백을 **반박**해 책임을 재배정 | **가맹점** 제출 자료 |
| 1CB (First Chargeback) 카드홀더레터 (보조) | 발급사 | 발급사가 카드홀더 대리 | 매입사에 거래대금을 **처음 청구** | 카드홀더 제출 자료 |

- **재반박:** "이 거래는 정당하다"를 가맹점 증빙(예: 탑승·운항 증빙, 3DS 인증기록, 거래 상세)으로 입증해 카드홀더의 분쟁을 되받습니다. VISA 규정상 **매입사 응답요건(§11.10.2.6, Table 11-94·11-95)** 을 근거로 합니다.
- **1CB:** 반대 방향입니다. 발급사가 카드홀더를 대리해 매입사에 처음 청구하는 절차로, **발급사 제출요건(§11.10.2.5)** 을 근거로 합니다.

### 방향 자동 감지 (반드시 먼저 판별)

VISA 접수내역(intake record)에는 **발급사(Issuer)·매입사(Acquirer) 필드**가 있습니다.
케이스를 받으면 **이 필드로 우리(하나카드)가 발급사인지 매입사인지 먼저 판별**하고, 그에 맞는 산출물을 만듭니다.

- 하나카드 = **매입사 → 인바운드 → 재반박(Acquirer Dispute Response)** ⭐
- 하나카드 = **발급사 → 아웃바운드 → 1CB 카드홀더레터**
- **기본 가정 = 인바운드(재반박).** 접수내역에서 방향이 불명확하면 인바운드로 보고, 담당자에게 방향 확인을 요청합니다.

## 2. 핵심 원칙 (반드시 지킴)

1. **사람이 최종 결정합니다 (Human-in-the-loop 필수).**
   - AI는 **초안·후보·제안만** 만듭니다.
   - **사유코드 최종 선택**과 **첨부서류 최종 승인**은 반드시 현업 담당자가 합니다.
   - AI 출력에는 항상 "확정 전 담당자 검토 필요"가 표시되어야 합니다.

2. **VISA 규정이 최우선 근거입니다.**
   - 사유코드, 증빙 가능/불가, 필수서류는 모두 `rules/` 폴더의 VISA 분쟁 규정을 근거로 합니다.
   - 규정에 없는 내용을 지어내지 않습니다. 근거가 약하면 "규정 확인 필요"라고 표시합니다.

3. **고객 서류는 매우 비정형입니다.**
   - JPG·PDF·이메일 등 형식이 제각각이고, 고객마다 내용·순서·용어가 다릅니다.
   - 형식에 의존하지 말고 "무슨 일이 언제 일어났나(사실·타임라인)"를 뽑아냅니다.

4. **개인정보(PII) 취급 — POC 전제.**
   - 이 골격은 검증용(POC)이라 **익명 더미 데이터만** 사용합니다.
   - 가짜 이름, 카드번호 `1111-1111-1111-1111` 등 마스킹 데이터만 넣습니다.
   - **실제 고객 PII는 절대 넣지 않습니다.** (실 운영 전 별도 보안 검토 필요)

5. **언어 규칙.**
   - 업무 규칙·에이전트 설명·분석 결과·체크리스트 → **한국어**
   - **분쟁 서류 본문(재반박서 / 1CB 카드홀더레터)만 영문**으로 작성합니다 (VISA 제출용).

## 3. 업무 흐름 (방향 판별 → 사유코드 → 필수증빙 → 재반박/1CB)

```
[VISA 접수내역 + 가맹점 자료 (인바운드) / 고객 제출 서류 (아웃바운드)]
        │
        ▼
 ① document-analyzer  ── 비정형 서류 읽기 → 요약·핵심사실·타임라인·쟁점
        │                  + 접수내역의 Issuer/Acquirer 필드로 방향 판별
        │                    (하나카드=매입사 → 인바운드/재반박, =발급사 → 아웃바운드/1CB)
        ▼
 ② rule-matcher       ── 사실 + VISA 규정(rules/) → 사유코드 확인 + 근거
        │                  + (방향에 맞는) 필수/유리 증빙 체크리스트
        │                    인바운드 = 매입사 응답요건(11-94/11-95), 아웃바운드 = 발급사 제출요건
   [현업 담당자: 사유코드·방향 최종 확인]  ← 사람 결정 지점
        │
        ▼
 ③ draft-writer       ── 확정 사유코드 + 사실 → ⭐재반박서(영문, 기본) 초안 + 증빙 목록
        │                  (아웃바운드면 1CB 영문 카드홀더레터)
   [현업 담당자: 서류 최종 승인]      ← 사람 결정 지점
```

- 각 단계는 `.claude/agents/` 의 서브에이전트가 담당합니다.
- 분쟁 서류 작성 세부 규칙(재반박·1CB 구조·톤·증빙)은 `.claude/skills/chargeback-1cb/SKILL.md` 에 있습니다.
  (폴더명은 `chargeback-1cb`지만, 내용은 **재반박 우선 + 1CB 보조** 양쪽을 다룹니다.)

## 이렇게 사용하세요 (실행 예시)

> **코딩은 필요 없습니다.** 터미널에서 이 폴더(`workshop/mvp/`)에서 `claude` 를 실행한 뒤,
> 아래처럼 **평소 쓰는 한국어 문장으로** 한 단계씩 시키면 됩니다.
> 각 단계 결과는 **사람이 직접 읽고 고친 다음** 다음 단계로 넘어갑니다 (사람이 최종 결정 — HITL).

**시작:** 터미널에서 아래를 입력합니다.

```
cd workshop/mvp
claude
```

이제 Claude Code 안에서 단계별로 아래 문장을 그대로 입력(복사·붙여넣기)하세요.

**1단계 — 서류 분석 + 방향 판별 (document-analyzer 활용)**

```
sample-data/case-anon-01 폴더의 서류를 분석해서
핵심 사실·타임라인·쟁점을 정리하고,
접수내역의 Issuer/Acquirer 필드로 이 건이 인바운드(매입사 방어)인지 판별해줘.
```

→ 결과로 "무슨 일이 언제 일어났는지" + **우리가 매입사인지 발급사인지(방향)** 가 정리됩니다.
→ **사람이 검토:** 사실관계·날짜·금액과 **방향 판별**이 맞는지 확인하고 틀린 부분은 고쳐서 알려줍니다.

**2단계 — 사유코드 확인 + (방향별) 필수증빙 (rule-matcher 활용)**

```
방금 정리한 사실과 방향(인바운드/매입사)으로 VISA 사유코드를 확인하고,
매입사 방어에 필요한 필수/유리 증빙 체크리스트를 근거와 함께 제시해줘.
```

→ 결과로 사유코드(예: 13.1)와 근거, **방향에 맞는 증빙 목록**(인바운드면 매입사 응답요건 11-94/11-95 기준)이 나옵니다.
→ **사람이 최종 결정:** 사유코드와 방향을 **확정**합니다. (← 사람 결정 지점)

**3단계 — 재반박서(영문) 초안 작성 (draft-writer 활용) — 기본**

```
확정한 사유코드 13.1로 매입사 재반박서(Acquirer Dispute Response) 영문 초안을 작성하고,
가맹점 증빙 목록도 함께 정리해줘.
```

→ 결과로 영문 재반박서 초안과 증빙 목록이 `sample-data/case-anon-01/output/representment-draft.md` 에 저장됩니다.
→ **사람이 최종 승인:** 모르는 값은 `[CONFIRM]`/`[MISSING]` 으로 표시되니, 담당자가 채우고 승인한 뒤 제출합니다. (← 사람 결정 지점)

> 🔁 **아웃바운드(하나카드=발급사) 건이면** 3단계에서 1CB 영문 카드홀더레터를 대신 만듭니다:
> "확정한 사유코드로 1CB 영문 카드홀더레터 초안을 작성해줘." → `output/1cb-draft.md` 에 저장됩니다.

> 💡 한 문장으로 이어서 시켜도 되지만(예: "1단계부터 3단계까지 진행해줘"),
> **워크숍에서는 단계마다 멈춰 사람이 확인**하는 방식을 권장합니다. 방향 판별과 사유코드 선택이 가장 중요한 판단이기 때문입니다.

## 4. 폴더 구조

```
workshop/mvp/
├── CLAUDE.md                       ← (이 파일) 업무 규칙
├── .claude/
│   ├── agents/                     ← 서브에이전트 3종
│   │   ├── document-analyzer.md    ① 서류 분석 + 방향 판별
│   │   ├── rule-matcher.md         ② 사유코드 + (방향별) 증빙
│   │   └── draft-writer.md         ③ 재반박서(기본)/1CB(보조) 초안 작성
│   └── skills/
│       └── chargeback-1cb/SKILL.md ← 분쟁 서류 작성 규칙 (재반박·1CB)
├── rules/
│   └── visa-rules.md               ← VISA 규정 (지금은 더미, 실 규정집으로 교체)
└── sample-data/
    ├── README.md                   ← 케이스 폴더 규칙
    └── case-anon-01/               ← 익명 더미 케이스 1건 (커밋됨)
        ├── (접수내역·가맹점 자료)
        └── output/                 ← AI 산출물 저장 위치
```
